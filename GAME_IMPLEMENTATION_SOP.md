# 棋類遊戲3.0 新增遊戲實作 SOP

本文件規範棋類遊戲3.0後續每款遊戲的完成方式。`蒐靈祭 Soulaween` 是目前的基準實作；新增遊戲應沿用相同完成標準，但不得複製遊戲專屬規則。

## 1. 基本原則

1. 一次只完成一款遊戲，完成後才開始下一款。
2. 規則不確定時先查既有程式碼、正式規則或可靠來源；仍無法確認就詢問，不得猜測。
3. UI、規則、AI、undo、日誌與響應式體驗必須一起完成。
4. 每款遊戲完成後更新本 SOP；至少經兩款遊戲驗證後，才抽成跨專案 Skill。

## 2. 網站與檔案入口

```text
index.html                         正式遊戲大廳，不放開發規劃
interface.html                     目前的共用遊戲介面與基準實作
assets/lobby.css                   遊戲大廳樣式
games/<game-id>/rules.html         該遊戲詳細規則
<GAME>_IMPLEMENTATION.md           該遊戲狀態、AI與驗證紀錄
GAME_IMPLEMENTATION_SOP.md         本文件
```

- 大廳視覺參考棋類遊戲2.0：淺色背景、遊戲卡片、每款遊戲的視覺預覽。
- 大廳資訊參考棋類遊戲1.0：遊戲名稱、簡介、玩家數、類型、特色與入口。
- 未完成遊戲可顯示介紹及規則，但不得標示為可遊玩。
- 遊戲頁的返回鍵必須回到 `index.html`。

## 3. 規則確認

1. 找出舊專案的規則程式、UI程式與初始資料。
2. 將程式實際行為與 `games/<game-id>/rules.html` 逐條比對。
3. 明確記錄配件、棋盤、初始設置、合法動作、回合階段、跳過規則、終局與勝負。
4. 規則頁與程式不一致時建立待確認項目，不直接選一個版本實作。

## 4. 遊戲資料契約

每款遊戲至少要能表達：

```text
state       完整且可序列化的遊戲狀態
move        一次合法操作的全部資料
turn        目前玩家
phase       move / select / collect / gameover 等階段
history     每一步操作前的完整 snapshot
logs        使用者可閱讀的行動紀錄
winner      未結束為 null，結束後為勝方
```

核心規則集中在可測試函式：

```text
createInitialState(options)
getLegalMoves(state)
applyMove(state, move)
getResult(state)
cloneState(state)
```

UI只顯示 state 並送出 move，不得另寫一套勝負規則。

## 5. 固定遊戲介面

### 頂欄

- 返回首頁
- 中文名與英文名
- 設計師與出版社
- 遊戲設定
- 手機高度約 60px，桌面窄版約 64px

### 勝率與分數

- 先手、後手 MCTS 勝率文字與雙色條
- MCTS計算期間保留上一組結果，不得閃回 `50:50`
- 新對局初始化為 `50:50`
- 終局勝方 `100%`、敗方 `0%`
- 分數卡、邊框、陰影與間距沿用 Soulaween

### 棋盤與操作

- 棋盤保持正方形與穩定尺寸
- 棋子或動作選擇區放在棋盤下方
- 玩家提示放在選擇區下方，單行顯示
- 保留開始新對局、遊戲說明、返回上一步

### 設定

- 先後手可選人類、隨機、電腦AI
- 手機玩家類型維持三欄並排
- AI提供簡單、中級、困難、專家與自訂迭代數
- 有開局變體時提供經典、隨機、指定棋盤
- 設定變更後清除舊AI統計並重新計算

### 資訊分頁

- 行動日誌：每一步、跳過、得分、undo與終局
- 規則說明：配件、設置、流程、特殊規則、勝利條件
- 遊戲介紹：遊戲、設計師、出版社

## 6. 響應式要求

- 手機優先，不得水平捲動
- 設定選項與主要操作優先並排，不因小螢幕直接改成直排
- 文字不得超出按鈕或遮住內容
- 玩家提示不得換行，必要時縮字或省略
- 桌面版可增加右側資訊，但不得放大棋盤破壞手機比例

## 7. MCTS與勝率

- 勝率只由MCTS產生，不使用人工啟發式
- AI決策使用完整設定迭代數
- 人類可操作局面的勝率使用設定迭代數一半，最低200、最高600次
- 同一次搜尋同時回傳最佳動作與統計，禁止重複搜尋
- UCT依節點玩家切換根玩家／對手視角
- 和局回傳0.5
- 根節點以訪問次數最多的動作決策
- MCTS在render後排程，計算期間保留上一組勝率
- 只有實測效能不足後，才加入Web Worker、樹重用或位元棋盤

## 8. 返回上一步

1. 每次玩家操作前保存完整snapshot。
2. 不寫反向規則，不逐欄位還原。
3. 人類對AI時回到上一個人類可操作狀態。
4. snapshot包含棋盤、分數、回合、階段、候選動作、日誌與AI統計。

## 9. 實作順序

1. 規則audit與待確認清單
2. 完成`rules.html`
3. 定義state、move、phase與result
4. 實作純規則及終局
5. 接入共用UI
6. 完成人類對人類
7. 完成snapshot undo與日誌
8. 完成隨機玩家
9. 完成MCTS、難度與勝率
10. 完成手機、桌面及操作驗證
11. 更新首頁卡片、實作文件與本SOP
12. commit、push並確認GitHub Pages

## 10. 驗證清單

### 規則與操作

- 初始設置正確
- 合法與非法動作正確
- 特殊階段、跳過、得分與終局正確
- 新對局、人類、AI、undo與日誌可運作

### AI與勝率

- 難度與自訂迭代生效
- 勝率只由MCTS產生且合計100%
- 計算期間不跳回50:50
- 落子後更新
- 終局勝方100%、敗方0%

### UI與發布

- 首頁直接呈現遊戲大廳，不呈現規劃頁
- 手機設定選項並排
- 玩家提示位置正確且不換行
- 無重疊、水平溢出或空白畫面
- 設定、首頁、資訊分頁可操作
- JavaScript無錯誤，`git diff --check`通過
- 本機HTTP操作通過，GitHub Pages回傳HTTP 200

## 11. 完成定義

規則一致、可從開局玩到終局、所有玩家類型可運作、MCTS／勝率／undo／日誌／設定完成、手機與桌面通過實測、文件與Git紀錄更新後，才算完成一款遊戲。
